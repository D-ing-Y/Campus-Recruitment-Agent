# v0.3 统一证据层与领域契约需求

状态：Proposed
日期：2026-07-15
关联 ADR：`docs/05_adr/0003-adopt-evidence-driven-dual-profile-architecture.md`

## 1. 目标

在不破坏 v0.1/v0.2 Runtime 和 LLM 基础设施的前提下，建立统一证据层和双画像领域契约，为后续候选人画像 Graph、岗位画像 Graph、RAG 和分布式存储提供稳定边界。

## 2. 范围

### 必须实现

- 本地 `EvidenceStore` repository 接口与 SQLite + 文件实现。
- `EvidenceArtifact`、`EvidenceFragment`、`EvidenceClaim`、`Provenance` schema。
- `CandidateProfile`、`CareerIntent`、`RoleProfile`、`GapAssessment` 首版 schema。
- `CapabilityOntology` 首版及别名规范化接口。
- 原始文件保存、文本保存、SHA-256 去重和证据查询。
- 从本地文本 fixture 提取 Claim 的结构化 LLM 流程。
- Claim 的 evidence 引用校验。
- Profile snapshot/version 的最小持久化契约。
- 证据 trace 与 eval。
- v0.1/v0.2 回归兼容。

### 非目标

- 不在 v0.3 接入真实招聘网站。
- 不解析复杂 PDF 版式和完整代码仓库。
- 不实现完整候选人画像循环和用户提问。
- 不实现向量检索、RAG、分布式数据库或 Multi-Agent。
- 不迁移到 Web UI。

## 3. 用户故事

- 作为系统，我可以保存一份原始简历或JD，并在后续解析器变化时重新处理原文。
- 作为开发者，我可以从一条 Claim 找到对应 Artifact 和原文 Fragment。
- 作为用户，我可以区分原文事实、我的自述和模型推断。
- 作为评估者，我可以统计不带证据的 Claim 比例和证据追溯率。
- 作为后续 Graph，我可以通过 repository 接口读取证据，而不依赖具体 SQLite 路径。

## 4. 功能要求

### FR-01 原始 Artifact 归档

- 支持本地 PDF、TXT、Markdown、HTML fixture 的登记。
- 保存原文件或复制到 `data/evidence/raw/`。
- 保存 `content_hash`、来源类型、内容类型、时间和非敏感 metadata。
- 相同内容重复导入时返回已有 artifact 或显式 deduplicated 结果。

### FR-02 文本与 Fragment

- 提取或接收标准化文本并保存到 `data/evidence/text/`。
- 生成 Fragment 并保存 locator。
- Fragment 必须可精确定位回 Artifact；暂不要求最佳语义切分。

### FR-03 Claim 提取

- 使用 v0.2 LLM Provider 和 structured output 能力从给定 Fragment 提取 Claim。
- Claim 必须包含 `evidence_fragment_ids`。
- 模型不得补充 Fragment 未表达的事实。
- 推断必须使用 `model_inference`，并记录 confidence 和模型/prompt/schema 版本。

### FR-04 Claim 校验

- 引用的 Fragment 必须存在且属于允许的 Artifact。
- `observed_fact` 和 `user_reported` 不允许 evidence 为空。
- Claim value 不满足 schema 时拒绝写入。
- Claim 更新通过新版本和 `supersedes_claim_id`，不原地覆盖历史。

### FR-05 领域契约

- CandidateProfile 与 RoleProfile 使用统一 capability ID。
- CareerIntent 与 CandidateProfile 分离。
- Profile snapshot 保存 supporting claim IDs、版本、生成时间和 schema 版本。
- GapAssessment 能表达四类差距，但 v0.3 不实现最终匹配算法。

### FR-06 Repository 抽象

- Agent/LLM 节点不得直接拼 SQL 或文件路径。
- repository 至少支持 artifact、fragment、claim 和 profile snapshot 的写入与按 ID 查询。
- 本地实现必须可被未来 PostgreSQL/S3 实现替换。

### FR-07 输出与追踪

- run 输出包含新建/复用的 evidence IDs、claim IDs、schema/prompt 版本和错误摘要。
- trace 不保存完整个人材料和模型密钥。

## 5. 非功能要求

- 默认测试使用匿名 fixture，不提交真实简历。
- 写入操作幂等；重复执行同一 fixture 不产生重复 Artifact。
- 原始文件写入成功后才提交对应元数据，失败时不得留下有效但不存在的 URI。
- SQLite schema 有显式版本和迁移入口。
- repository 单元测试不依赖网络或真实 LLM。

## 6. Eval 指标

- `artifact_dedup_rate`
- `claim_schema_valid_rate`
- `evidence_trace_rate`
- `unsupported_claim_rate`
- `fragment_locator_valid_rate`
- `profile_claim_reference_valid_rate`
- v0.1/v0.2 regression pass rate

## 7. 验收标准

- 给定匿名简历、项目说明和JD fixture，可完成 Artifact→Fragment→Claim 全流程。
- 100% `observed_fact` 和 `user_reported` Claim 可定位到原始 Fragment。
- 故意生成无引用 Claim 时，校验器拒绝写入并产生可读错误。
- 重复导入相同文件不产生重复 Artifact。
- 可保存和读取一个 CandidateProfile、CareerIntent 和 RoleProfile snapshot。
- 旧 CLI smoke test 继续可运行，v0.1/v0.2 测试不因新模块失效。
