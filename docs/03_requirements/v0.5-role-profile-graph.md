# v0.5 岗位需求画像 Graph 需求

状态：Accepted / Ready for Implementation
日期：2026-07-18
关联 ADR：`docs/05_adr/0005-separate-source-channels-and-role-profile-levels.md`
关联 RFC：`docs/04_rfc/0005-role-profile-graph.md`

## 1. 背景

v0.4 已实现可恢复的候选人画像 Graph，并验证了 Evidence Store、ProfileSnapshot、
ToolRegistry、结构化 LLM、conditional edge、loop、SQLite checkpoint 和
interrupt/resume。下一步需要以同样的证据原则构建岗位需求画像。

岗位信息与候选人材料存在两个关键差异：

- 岗位事实来自持续变化的外部网页，必须记录查询、来源、发布时间、获取时间和失效状态；
- 招聘岗位页与社区面经承担不同证据职责，不能混为同等权威的一份“岗位总结”。

v0.5 要形成项目第二个真实业务 subgraph：围绕一个岗位方向动态规划查询、采集并先归档原始网页、归一化与去重、生成具体岗位画像和岗位族画像、评价证据覆盖，再决定继续搜索、换词、换源、等待用户授权或停止。

## 2. 目标

实现以下可恢复闭环：

```text
CareerIntent / SearchScope
  → 查询规划
  → 招聘来源与经验来源分别采集
  → 原始响应先归档为 EvidenceArtifact
  → 文本提取与结构化归一化
  → 跨来源去重
  → Role Claim 提取与校验
  → JobInstanceRoleProfile
  → RoleFamilyProfile 聚合
  → 覆盖度与时效性评价
  → 继续搜索 / 换词 / 换源 / 等待授权 / 保留 unknown / 完成
```

## 3. 范围

### 3.1 必须实现

- 独立 `role_profile` LangGraph subgraph。
- `SearchScope`、`RoleQueryPlan`、`SourceQuery`、`SourceRunReceipt` 契约。
- 招聘来源与经验来源分离的 adapter、归档、归一化和证据策略。
- 首版真实招聘来源 adapter：`zhaopin_jobs`，目标入口为智联校园。
- 首版真实经验来源 adapter：`nowcoder_experience`，目标为牛客讨论/面经内容。
- 用户正常登录和本地 cURL/Cookie 导入的授权边界；不得自动绕过验证。
- 离线默认 fixture adapter，不依赖实时网络或登录。
- `NormalizedJobPosting`、`ExperienceEvidenceRecord` 和跨来源去重集群。
- `JobInstanceRoleProfile` 与 `RoleFamilyProfile` v0.5。
- 硬性资格、核心能力、加分项、职责、工作场景、公司特异项和招聘筛选信号分层。
- Capability Ontology 规范化；未知技能保留 raw label。
- `RoleCoverageAssessment` 与查询/来源覆盖缺口。
- `search_more`、`change_query`、`change_source`、`await_user_auth`、
  `finalize_with_unknowns`、`complete`、`fail` 条件路由。
- SQLite checkpoint、授权 interrupt/resume、循环与调用预算。
- 所有 Role/Profile 事实保存 supporting Claim、source URL、published/retrieved time 和 confidence。
- trace、SourceRunReceipt、fixture、评估指标和 v0.1-v0.4 回归。

### 3.2 真实来源与离线验收

- live collector 必须通过 SourceAdapter 和 ToolRegistry 调用，不在 Graph 节点硬编码平台请求。
- live run 必须先保存原始 HTML/JSON/text，再进行归一化或模型总结。
- live 网络 smoke 由用户在本地显式启用，不进入默认 CI。
- 登录受限时，系统返回 `authentication_required`，由用户在真实 Chrome 正常登录后把
  Copy as cURL 内容导入本地凭据文件；Graph State、trace、Evidence metadata 和报告只保存
  `credential_ref`，不得保存 Cookie 或 Authorization 值。
- 可提交测试使用 L0 合成数据和 L1 匿名/最小化 fixture；允许保存的 L2 公开快照必须记录来源和获取时间，且不得批量转载受限制内容。
- 每个 live adapter 完成条件包括一次本地 opt-in smoke 和不含密钥的 SourceRunReceipt。

### 3.3 必须保持的证据边界

- 岗位是否存在、城市、职责、资格、申请入口和截止时间只能由企业官方页或招聘岗位页确认。
- 社区面经只能形成 `hiring_signal`、`experience_signal` 或低权威推断，不能确认岗位仍在招聘，也不能单独创建 hard requirement。
- 面经的公司级、岗位族级、具体岗位级适用范围必须显式保存；未知范围不得强行归到具体岗位。
- 原始网页是事实源，Normalized Record、Claim 和 RoleProfile 都是可重建派生状态。
- ProfileSnapshot、checkpoint、聊天摘要和模型总结都不能替代原始来源。

### 3.4 与 CareerIntent、CandidateProfile 的边界

- `CareerIntent` 提供目标岗位、城市、行业、公司范围、毕业年份和 hard scope。
- v0.5 不使用 CandidateProfile 能力筛除岗位，也不计算候选人与岗位的匹配度。
- 搜索阶段只排除明确违反 hard scope 的记录，且保存排除原因；偏好型过滤和用户选择留给 v0.6。
- 相关但不完全匹配的岗位不得静默丢弃，应保留 `included`、`deferred` 或
  `excluded_hard_scope` 状态。

### 3.5 非目标

- 不实现 CandidateProfile 与 RoleProfile 的覆盖百分比、差距解释或 Offer 概率。
- 不实现学习路线、笔面试准备计划或自动投递。
- 不实现向量检索、Hybrid RAG、embedding 或 reranker。
- 不实现分布式存储、远程队列、Multi-Agent 或 Web UI。
- 不实现验证码绕过、设备指纹伪造、代理池或攻击性反爬。
- 不要求覆盖 Boss、小红书、脉脉等全部来源；它们只用 fixture 验证 adapter contract 或留待后续。
- 企业官网不是首版岗位发现核心；只有用户指定公司或已有确切岗位 URL 时才作为验证/补充来源。

## 4. 输入与输出

### 4.1 输入

- `run_id`、`thread_id`、`user_id`。
- `career_intent_snapshot_id` 或已校验 `SearchScope`。
- 启用的 recruitment/experience source IDs。
- 可选的指定公司、已知岗位 URL 或本地原始快照。
- source、query、document、LLM 和 Tool 预算。
- resume 时的 `credential_ref`、跳过来源或取消动作，不包含凭据正文。

### 4.2 输出

- `job_posting_ids` 和 `job_cluster_ids`。
- 0..N 个 `job_instance_profile_snapshot_ids`。
- 0..1 个当前搜索范围的 `role_family_profile_snapshot_id`。
- `RoleCoverageAssessment` 与未解决 coverage gaps。
- `SourceRunReceipt` 列表和 supporting artifact/fragment/claim IDs。
- Graph 状态：`completed`、`completed_with_unknowns`、`interrupted` 或 `failed`。
- trace、LLM/tool 调用、来源警告、预算消耗和错误摘要。

## 5. 功能要求

### FR-01 SearchScope 与查询规划

- SearchScope 必须从已校验 CareerIntent 或显式输入构建。
- 每个 role graph run 只处理一个 canonical target role family；CareerIntent 中存在多个方向时由调用方启动多个独立 run。
- 至少包含目标岗位关键词、canonical role family、城市、毕业年份、招聘类型和 hard scope。
- QueryPlan 分别生成 recruitment 与 experience 查询，不把两类结果混在一个 adapter。
- query 必须有稳定 `query_id`/fingerprint，可记录换词原因和父 query。
- 模型可以建议同义词和查询分解，确定性 policy 校验 source 能力、预算和 hard scope。

### FR-02 SourceAdapter 与真实采集

- recruitment/experience adapter 遵守相同生命周期接口，但输出不同 schema。
- collector 返回任何可解析内容前必须先归档原始响应。
- 归档 metadata 至少包含 source ID、channel、query ID、source URL、HTTP/status、
  published/retrieved time、content hash、parser version 和 warnings。
- 分页/cursor、限速、空结果、登录要求、来源变化和可重试错误必须结构化表达。
- 不得把 Cookie、完整请求头、cURL 或 API key 写入 ToolResult、Artifact metadata、State 或 trace。

### FR-03 招聘岗位归一化

- 归一化字段遵守 `NormalizedJobPosting` 契约。
- 未知字段使用 `unknown`、null 或空数组，不从常识补齐。
- 保留原始职责、原始要求、标准化要求、申请 URL、来源 URL 和时间。
- `status=excluded_hard_scope` 只能用于明确违反 hard scope，并保存机器可读原因。
- 不因标题噪声、薪资缺失或描述不完整静默丢弃岗位。

### FR-04 经验帖归一化

- 经验帖先保存原始 HTML/text，再生成严格 JSON。
- 保留平台、URL、发布时间、获取时间、公司、岗位、城市、阶段和适用范围。
- 提取 written exam、interview、tech stack、project preference 等 signal。
- 每个 signal 必须引用 EvidenceFragment；无支撑内容留空或 unknown。
- 公司级内容不得标为具体岗位事实；跨公司/跨岗位内容不得合并成单条记录。

### FR-05 去重与冲突

- 相同 source URL/content hash 的记录幂等复用。
- JobPosting 先用 URL/hash 或包含招聘周期与内容签名的确定性 identity key 聚类；仅标题/公司相似只能生成 merge candidate，不得由 LLM 直接合并。
- 跨平台重复岗位保留全部 source refs，但在岗位族统计分母中只计一个 job instance。
- 经验帖转载/重复 hash 只计一个独立 signal source。
- 来源字段冲突时按 authority policy 选择 confirmed 值并保留 conflicting claims，不能覆盖历史。

### FR-06 Role Claim 与具体岗位画像

- Role Claim 只能从已归档 Fragment 提取并经过 ClaimValidator。
- `JobInstanceRoleProfile` 必须区分硬性资格、职责、核心能力、加分项、工作场景和公司特异项。
- 每个事实字段保存 supporting claim IDs、来源权威级别、freshness 和 confidence。
- 面经 signal 单独挂载，不反向改写官方资格要求。
- 失效、过期或无法验证的岗位保留 snapshot 与 status，不删除历史。

### FR-07 岗位族画像

- 岗位族画像从去重后的具体岗位画像和去重后的经验 signal 聚合。
- 聚合必须保存样本定义、独立岗位数、公司数、地点、时间窗口和分母。
- requirement prevalence、公司覆盖和 signal frequency 由确定性代码计算。
- `common`/`frequent`/`observed` 分层必须使用配置化阈值；样本不足时禁止称为“普遍要求”。
- 公司特异项和地域/时间特异项不得提升为岗位族通用要求。
- LLM 可解释聚合结果，不得自由修改计数、分母或阈值。

### FR-08 覆盖度评价与动态路由

- 覆盖度至少评价招聘字段完整度、有效岗位数量、公司多样性、来源权威、时间新鲜度、经验 signal 覆盖和冲突。
- 有可继续翻页或新关键词且预算允许时选择 `search_more` 或 `change_query`。
- 当前来源不可用、权威不足或覆盖缺口需要其他来源时选择 `change_source`。
- 必需来源需要用户正常登录时选择 `await_user_auth`。
- 预算耗尽、用户跳过或边际信息价值过低时选择 `finalize_with_unknowns`。
- 达到覆盖标准且无阻塞冲突时选择 `complete`。
- 路由建议必须经过确定性预算与来源 policy。

### FR-09 授权中断与恢复

- 复用 Human Interaction contract，新增 `authorize_source` interaction type。
- request 只包含 source、登录入口、导入说明和期望 credential ref，不包含 Cookie。
- 用户在 Graph 外完成正常登录和本地凭据导入，再以 `authorized` + `credential_ref` resume。
- `skip_source` 后不得重复请求同一来源；Graph 换源或以 unknown 完成。
- 错误 thread/request/source/credential ref 必须拒绝，且 Evidence Store 零写入。

### FR-10 Checkpoint、幂等与可观测性

- 复用 v0.4 SQLite checkpointer，并为 role thread 使用独立 thread ID。
- query、source batch、raw artifact、normalized record、Claim 和 ProfileSnapshot 都必须有稳定幂等键。
- 进程重启后可从收集循环或授权中断恢复。
- trace 记录 query/source/record 数量、路由原因、时效覆盖和错误类型，不记录完整网页或凭据。

## 6. 默认覆盖与预算基线

默认值用于 deterministic baseline，可由配置降低：

```text
max_query_rounds = 3
max_queries = 12
max_source_switches = 2
max_documents = 60
max_llm_calls = 20
max_tool_calls = 50
```

岗位族画像称为“具备基础样本”至少需要：

- 3 个去重后的有效/近期 job instance；
- 覆盖至少 2 家公司；
- 关键招聘字段均有招聘/官方来源 Claim；
- 已尝试至少一个 experience source。

若真实市场没有足够岗位，系统必须以 `insufficient_sample` 和 unknown 完成，而不是无限搜索或降低证据门槛。

经验 signal 至少需要 2 个去重后的独立帖子才能输出频率描述；单帖只作为
`observed_signal` 展示。

## 7. 非功能要求

- 默认全量测试不访问网络、不需要登录或真实 LLM。
- live adapter 必须可通过 config 禁用，并遵守限速、超时、重试上限和来源使用条款。
- 网络失败不能破坏已经归档的 Artifact 或已完成 snapshot。
- 原始 live 数据、Cookie/cURL、credential store、checkpoint 和运行报告默认进入 Git 忽略目录。
- Schema、prompt、parser、adapter 和 source policy 必须版本化。
- 真实来源结构变化必须返回 `source_changed`，不得静默生成错误字段。

## 9. 验收标准

- 给定一个 AI Agent/LLM 应届岗位 SearchScope，可从 fixture 生成若干具体岗位画像和一个岗位族画像。
- 招聘与经验来源的原始响应均先归档，再归一化、提取 Claim 和构建画像。
- 100% 事实性 Role 字段可回溯到允许该字段的来源 Fragment。
- 社区面经不能单独确认岗位存在、硬性资格或申请截止时间。
- 跨平台重复岗位只在岗位族分母中计一次，所有来源仍可追溯。
- 查询覆盖不足时 Graph 能换词/换源；达到预算后显式以 insufficient sample 结束。
- 授权来源能 interrupt/resume，Cookie/cURL 不进入 State、trace、Evidence 或 Git。
- `zhaopin_jobs` 与 `nowcoder_experience` 各完成一次本地 opt-in live smoke，并生成无凭据 SourceRunReceipt；默认 CI 仍为离线。
- SQLite checkpoint 跨 Graph 实例恢复且重复 source batch 不重复写入。
- v0.1-v0.4 全量回归通过，并生成真实 v0.5 eval report 后才可标记 Implemented。
