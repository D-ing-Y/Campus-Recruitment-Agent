# v0.4 候选人画像 Graph 需求

状态：Implemented
日期：2026-07-17
实现验收：2026-07-18，68 项测试通过
关联 ADR：`docs/05_adr/0004-use-stateful-candidate-profile-subgraph.md`
关联 RFC：`docs/04_rfc/0004-candidate-profile-graph.md`

## 1. 背景

v0.3 已实现 `EvidenceArtifact → EvidenceFragment → EvidenceClaim → ProfileSnapshot`
统一证据链，但证据管线仍以独立模块和测试入口运行，尚未形成可暂停、可恢复的候选人画像闭环。

v0.4 要把 v0.1 的 LangGraph Runtime、v0.2 的结构化 LLM 调用和 v0.3 的统一证据层连接起来，形成项目第一个具有条件路由、循环、持久化 checkpoint 和 human-in-the-loop 的业务 subgraph。

## 2. 目标

实现以下可恢复闭环：

```text
提交候选人材料
  → 归档与解析
  → 提取并校验 Claim
  → 构建初始 CandidateProfile
  → 评价信息充分性
  → 继续读取 / 定向提问 / 请求材料 / 保留 unknown
  → 用户回答或纠正先写入证据层
  → 重建并版本化 CandidateProfile
  → 完成或在预算内继续循环
```

## 3. 范围

### 3.1 必须实现

- 独立 `candidate_profile` LangGraph subgraph。
- 可读取文本型 PDF、Markdown、TXT 和项目 README 的真实本地工具。
- 复用 v0.3 ingestion、fragment、ClaimValidator、repository 和 projector。
- `CandidateProfile` v0.4、`SufficiencyAssessment`、`InformationGap`、`QuestionPlan` 契约。
- 基于信息价值的高价值缺口排序。
- `read_more`、`ask_user`、`request_more_materials`、`finalize_with_unknowns`、`complete` 条件路由。
- 本地持久化 checkpoint；测试可使用内存 checkpointer。
- LangGraph `interrupt()` 与相同 `thread_id` 的 resume。
- 用户回答、补充说明和纠正先保存为 Artifact/Fragment/Claim，再参与画像重建。
- 画像 snapshot 版本递增、版本比较和 supporting claim 引用。
- 循环次数、问题数量、LLM 调用和工具调用预算。
- trace、结构化错误、评估 fixture 和 v0.1-v0.3 回归。

### 3.2 必须保持的边界

- `CareerIntent` 与 `CandidateProfile` 分离；岗位、城市、薪资和行业偏好不参与候选人能力充分性评分。
- State 只保存引用、小型结构化决策和计数器，不保存完整原文或二进制。
- LLM 不能直接写入确认画像；输出必须经过 Pydantic、ClaimValidator 和 repository。
- 用户回答不能只留在聊天记录或 checkpoint 中，必须先证据化。
- 无法证明的能力保留为 `unknown`、`inferred` 或 `conflicted`，不得补全。

### 3.3 非目标

- 不实现岗位检索、岗位需求画像、双画像匹配和学习计划。
- 不接入真实招聘网站、社区爬虫或浏览器自动化。
- 不实现 OCR、表格还原、多栏复杂版式和完整代码仓库静态分析。
- 不实现向量检索、Hybrid RAG、分布式存储或 Multi-Agent。
- 不实现 Web UI；CLI 或测试驱动的最小 interrupt/resume 入口即可。
- 不要求外部 MCP 服务器；v0.4 的文档处理和存储使用仓库内真实本地 Tool。

## 4. 输入与输出

### 4.1 输入

- `thread_id`、`run_id`、`user_id`、`candidate_id`。
- 本地文件路径或已归档 `artifact_id`。
- 可选的 `CareerIntent` 引用或结构化表单数据。
- resume 时的用户回答、补充文件、纠正、跳过或确认动作。
- Graph 预算与本地存储配置。

### 4.2 输出

- 最新 `candidate_profile_snapshot_id`。
- `CandidateProfile` 小型摘要与版本号。
- supporting `artifact_ids`、`fragment_ids` 和 `claim_ids`。
- `SufficiencyAssessment` 与仍未解决的 `InformationGap`。
- Graph 状态：`completed`、`completed_with_unknowns`、`interrupted` 或 `failed`。
- trace、LLM call、tool call、checkpoint 和错误摘要。

## 5. 功能要求

### FR-01 材料摄取

- PDF、Markdown、TXT 和常见 README 文件必须通过 `ToolRegistry` 调用真实实现。
- 文本型 PDF 必须提取页码可定位文本；扫描件或无文本 PDF 返回结构化 `unsupported_input`，并允许路由到请求其他格式。
- 重复文件复用 v0.3 SHA-256 去重结果。
- 原文件必须先归档，后续解析、分片和 Claim 提取只引用已归档 Artifact。

### FR-02 Claim 提取与投影

- 复用 v0.2 `parse_structured_output()` 和 v0.3 Claim 提取/校验路径。
- Claim 必须引用当前用户有权访问的 Fragment。
- `CandidateProfile` 只投影已持久化且状态有效的 Claim。
- 每次有效 Claim 集变化后生成新 snapshot；相同输入重放不得产生语义重复版本。

### FR-03 充分性评价

- 评价至少覆盖教育、经历、能力、责任边界和证据质量。
- 每个缺口包含目标字段、原因、重要性、可回答性、推荐获取方式和关联证据。
- 充分性是“能否形成诚实、有证据边界的当前画像”，不是“是否满足某个岗位”。
- 不允许仅用单个总分隐藏关键冲突或高影响未知项。

### FR-04 动态路由

- 有未处理且可能补足当前缺口的材料时优先 `read_more`。
- 缺口适合用户直接回答且问题预算允许时选择 `ask_user`。
- 缺口需要可核验文件或当前格式不可解析时选择 `request_more_materials`。
- 达到预算、用户跳过或继续获取信息的预期价值过低时选择 `finalize_with_unknowns`。
- 关键画像信息已经足够且无阻塞冲突时选择 `complete`。
- 路由结果必须经过枚举校验和确定性预算守卫。

### FR-05 人工中断与恢复

- 中断载荷必须符合 `HumanInteractionRequest`，含稳定 `request_id`、原因、问题、目标字段和证据引用。
- 相同 `thread_id` 恢复；错误或过期 `request_id` 必须拒绝。
- 用户回答、纠正和上传文件先由 `archive_human_input` 节点证据化。
- 重复提交同一 resume 输入不得产生重复 Artifact、Claim 或重复画像版本。
- 进程重启后可从本地 checkpoint 恢复到中断点。

### FR-06 纠正与冲突

- 用户纠正生成新的 `user_reported` Claim，并通过 `supersedes_claim_id` 关联被纠正 Claim。
- 原 Claim 保留历史并变为 `superseded`，不得原地删除。
- 若纠正与材料事实冲突，画像必须标为 `conflicted`，而不是静默选择一方。
- 画像版本比较至少能列出新增、移除、改变和仍有冲突的字段。

### FR-07 CareerIntent 分离

- v0.4 可保存和读取独立 `CareerIntent` snapshot。
- 用户修改求职偏好不得触发候选人能力 Claim 的变更。
- 候选人画像充分性评价不得把薪资、城市或目标岗位缺失计为能力画像不足。

### FR-08 可观测性

- 每个节点记录开始/结束时间、状态、输入/输出摘要和错误类型。
- trace 不记录完整简历、完整用户回答、密钥或绝对隐私内容。
- 每轮记录路由原因、预算消耗、snapshot 版本和 supporting claim 数量。

## 6. 非功能要求

- 本地默认实现不依赖网络或真实 LLM。
- Graph 节点只通过 repository、ToolRegistry 和 LLM Provider 抽象访问外部能力。
- checkpoint 与 Evidence Store 使用不同职责的持久层，checkpoint 不能成为事实源。
- 所有写操作必须有稳定幂等键。
- 单次 run 默认最多 3 个画像循环、每次中断最多 3 个问题；具体默认值由配置定义。
- 达到硬预算后必须终止或中断，不得无限循环。
- 真实个人材料、API key、Cookie 和 checkpoint 数据库不得提交 Git。

## 7. Eval 指标

- `candidate_route_accuracy`
- `high_value_gap_recall`
- `question_actionability_rate`
- `redundant_question_rate`
- `interrupt_resume_success_rate`
- `checkpoint_recovery_rate`
- `profile_evidence_coverage_rate`
- `profile_correction_trace_rate`
- `resume_idempotency_violation_count`
- `max_loop_termination_rate`
- v0.1-v0.3 regression pass rate

## 8. 验收标准

- 信息充分 fixture 可在无人工中断下生成带证据引用的 CandidateProfile。
- 信息不足 fixture 能识别 gold 高价值缺口并产生不超过预算的定向问题。
- 用户回答恢复后，回答先成为 Artifact/Fragment/Claim，再生成新 profile snapshot。
- 用户纠正可回溯到新旧 Claim，旧 snapshot 保留且版本差异可读。
- 用户跳过、扫描件无法解析或预算耗尽时，以 unknown 明确完成，不发生无限循环。
- 使用持久化 checkpointer 时，重启后以相同 `thread_id` 成功恢复。
- 重复 resume 不产生重复证据或画像版本。
- v0.1-v0.3 全量回归通过，并生成 v0.4 eval report。
