# v0.1 Mini Agent Runtime 需求文档

状态：Implemented
日期：2026-07-07
负责人：DingYi
实现入口：`campus-job-agent/`

## 1. 背景

本项目已从“秋招检索 Skill + 爬虫脚本”重构为面向 2027 秋招的垂直领域 Job Intelligence Agent。v0.1 的目标不是接入真实招聘网站，也不是实现完整业务能力，而是先建立一个可运行、可测试、可扩展的 Mini Agent Runtime。

v0.1 必须证明项目具备 Agent 工程的基础结构：

- 有明确的状态对象。
- 有可执行的工作流图。
- 有工具注册与调用机制。
- 有结果校验。
- 有执行轨迹。
- 有最终报告输出。

## 2. 目标

v0.1 交付一个基于 LangGraph 的最小闭环 Agent Runtime。

最小闭环：

```text
用户目标
  -> 目标解析
  -> 计划生成
  -> mock 工具调用
  -> 结果校验
  -> 报告生成
```

必须实现：

1. 使用 CLI 接收一个用户目标文本。
2. 将用户目标解析为结构化 `ParsedGoal`。
3. 根据 `ParsedGoal` 生成一个可执行计划 `Plan`。
4. 通过 `ToolRegistry` 调用至少一个 mock tool。
5. 将工具结果写入 `AgentState`。
6. 使用 `Verifier` 校验关键字段是否存在。
7. 记录每个节点的执行 trace。
8. 输出 Markdown 报告和 JSON trace。
9. 提供单元测试和集成测试。

## 3. 非目标

v0.1 不做以下内容：

- 不接入 Boss、智联、牛客、小红书等真实网站。
- 不接入真实 LLM API。
- 不实现 RAG、向量数据库或长期记忆。
- 不实现 Evidence Store 的完整持久化能力。
- 不实现多 Agent 或 Sub-Agent。
- 不实现 Web UI 或 Streamlit。
- 不实现服务器部署。
- 不处理真实 cookie、登录态、验证码或反爬问题。
- 不实现完整 Skill 引擎。

## 4. 用户输入

CLI 输入一个自然语言目标。

示例：

```bash
campus-agent run "成都 AI Agent 2027 秋招"
```

v0.1 只要求支持单轮命令式运行，不要求支持持续对话。

## 5. 预期输出

每次运行生成一个 `run_id`，并在本地输出：

```text
data/runs/<run_id>/state.json
data/runs/<run_id>/trace.json
data/reports/<run_id>.md
```

CLI 输出至少包含：

- `run_id`
- 运行状态
- 报告路径
- trace 路径

## 6. 核心工作流节点

v0.1 工作流必须包含 5 个节点。

### 6.1 `parse_goal`

输入：

- `user_input`

输出：

- `parsed_goal`

最低解析字段：

- `role_query`
- `city`
- `graduation_year`
- `raw_text`

对于无法解析的字段，使用 `unknown`。

### 6.2 `plan_tasks`

输入：

- `parsed_goal`

输出：

- `plan`

最低计划内容：

- 至少 1 个 task。
- 每个 task 包含 `task_id`、`tool_name`、`args`、`reason`。
- v0.1 默认生成一个调用 `mock_job_search` 的任务。

### 6.3 `run_mock_tool`

输入：

- `plan`

输出：

- `tool_results`

要求：

- 通过 `ToolRegistry` 调用工具。
- 不允许节点直接调用 mock tool 函数。
- mock tool 返回固定但结构化的数据。
- 工具调用失败时必须记录错误，不允许静默失败。

### 6.4 `verify_result`

输入：

- `parsed_goal`
- `plan`
- `tool_results`

输出：

- `verification`
- `errors`

最低校验项：

- `parsed_goal.role_query` 存在。
- `plan` 非空。
- 至少有一个 `tool_result`。
- 每个 `tool_result` 包含 `tool_name`、`status`、`records`、`error` 字段。

### 6.5 `write_report`

输入：

- `parsed_goal`
- `plan`
- `tool_results`
- `verification`
- `trace`

输出：

- `report_path`

报告必须包含：

- 用户目标。
- 解析结果。
- 执行计划。
- mock 工具结果摘要。
- 校验结果。
- trace 摘要。

## 7. 状态对象范围

v0.1 `AgentState` 至少包含：

```json
{
  "run_id": "string",
  "user_input": "string",
  "parsed_goal": {},
  "plan": [],
  "tool_results": [],
  "verification": {},
  "trace": [],
  "errors": [],
  "report_path": null
}
```

实现时可使用 Pydantic model 或 TypedDict。若 LangGraph 节点对状态合并有特殊要求，以 RFC 设计为准。

## 8. Tool Contract 范围

v0.1 至少实现一个工具：

```text
mock_job_search
```

工具输入：

```json
{
  "role_query": "AI Agent",
  "city": "成都",
  "graduation_year": "2027"
}
```

工具输出：

```json
{
  "tool_name": "mock_job_search",
  "status": "success",
  "records": [
    {
      "company": "Mock Company",
      "role_title": "AI Agent Intern",
      "city": "成都",
      "requirements": ["Python", "LLM", "Agent"]
    }
  ],
  "evidence_ids": [],
  "error": null,
  "metadata": {}
}
```

v0.1 的 `evidence_ids` 可以为空，但字段必须存在。

## 9. Trace 要求

每个节点执行后必须写入一条 trace。

单条 trace 至少包含：

```json
{
  "node": "parse_goal",
  "status": "success",
  "started_at": "ISO-8601",
  "ended_at": "ISO-8601",
  "duration_ms": 12,
  "input_summary": {},
  "output_summary": {},
  "error": null
}
```

Trace 目标：

- 支持调试。
- 支持后续 eval。
- 支持报告摘要。

## 10. 错误处理要求

v0.1 不要求复杂恢复，但必须做到：

- 节点异常不能被静默吞掉。
- 异常必须进入 `errors`。
- trace 中必须记录失败节点。
- CLI 退出时必须返回可读错误信息。
- 若 `verify_result` 不通过，仍允许生成报告，但报告中必须标记失败原因。

## 12. 测试要求

### 12.1 单元测试

至少覆盖：

- goal parsing。
- plan generation。
- tool registry 注册与调用。
- mock tool 输出结构。
- verifier 成功与失败情况。

### 12.2 集成测试

至少覆盖：

- 输入 `"成都 AI Agent 2027 秋招"`。
- 完整 graph 运行成功。
- 生成 `state.json`、`trace.json`、Markdown report。
- trace 中包含 5 个核心节点。

### 12.3 Eval 检查

v0.1 eval 可以是规则检查，不需要模型参与。

最低 eval：

- `graph_completed = true`
- `required_nodes_present = true`
- `mock_tool_called = true`
- `report_generated = true`
- `verification_passed = true`

## 13. 验收标准

v0.1 完成必须满足：

1. 可以通过 CLI 跑通一次完整流程。
2. 输出 `run_id`、报告路径和 trace 路径。
3. `state.json` 存在且包含核心状态字段。
4. `trace.json` 存在且包含 5 个核心节点。
5. Markdown report 存在且内容可读。
6. 单元测试通过。
7. 集成测试通过。
8. v0.1 eval 检查通过。
9. README 或开发文档中补充运行方式。
10. 不引入真实招聘数据、cookie 或 API key。

## 14. 后续扩展接口

v0.1 需要为后续版本保留扩展点：

- v0.2：将 `parse_goal` 或 `plan_tasks` 替换为 LLM API 驱动。
- v0.3：将 `mock_job_search` 替换为真实工具接口。
- v1.0：接入招聘检索工具。
- v2.0：接入 Evidence Store、Memory 和 RAG。
- v2.5：拆分 Sub-Agent。

## 15. 未决问题

以下问题不阻塞 v0.1，但需要在 RFC 或后续版本中继续确认：

- `AgentState` 使用 Pydantic model 还是 TypedDict。
- CLI 是否直接放在 `apps/cli/main.py`，还是通过 package console script 暴露。
- trace 是否先写 JSON 文件，还是后续统一进入 SQLite。
- LangGraph 是否在 v0.1 引入 checkpoint，还是 v0.2/v0.3 再做。
