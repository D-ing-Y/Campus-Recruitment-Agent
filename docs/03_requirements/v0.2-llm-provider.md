# v0.2 LLM Provider 与结构化输出需求文档

状态：Implemented
日期：2026-07-08
负责人：DingYi
实现入口：`campus-job-agent/`

## 1. 背景

v0.1 已经建立基于 LangGraph 的 Mini Agent Runtime，完成了状态传递、规则目标解析、计划生成、mock 工具调用、结果校验、trace 和报告输出。

v0.2 的目标是在不扩大业务范围的前提下，引入真实 LLM API 调用能力，并把 v0.1 的规则解析逐步升级为可校验、可重试、可缓存的结构化 JSON 输出。

本阶段重点不是让 Agent 变得“会聊天”，而是建立后续岗位抽取、经验帖总结、能力包生成所依赖的模型层工程基础：

- 有统一的 LLM Provider 抽象。
- 有 OpenAI-compatible API 实现。
- 有 JSON-only prompt contract。
- 有 Pydantic schema 校验。
- 有有限重试和错误记录。
- 有本地缓存，避免相同输入重复调用。
- 有 LLM 调用 trace 元信息。

## 2. 目标

v0.2 交付一个可配置、可测试、可观测的 LLM Provider 层，并将 `parse_goal` 节点升级为优先使用 LLM 结构化解析。

最小闭环：

```text
用户目标
  -> LLM JSON 目标解析
  -> Pydantic 校验
  -> 校验失败时有限重试
  -> 生成结构化 SearchGoal / ParsedGoal
  -> 计划生成
  -> mock 工具调用
  -> 结果校验
  -> 报告生成
```

必须实现：

1. 定义 `LLMProvider` 抽象，屏蔽具体 API provider 差异。
2. 实现一个 OpenAI-compatible provider。
3. 支持从 `.env` 或本地 config 读取 provider、base_url、model、api_key。
4. 定义 v0.2 目标解析 schema，命名为 `SearchGoal` 或升级版 `ParsedGoal`。
5. 为目标解析提供 JSON-only prompt contract。
6. 使用 Pydantic 校验 LLM 输出。
7. 当输出不是合法 JSON 或 schema 校验失败时，执行有限重试。
8. 实现 LLM 调用缓存，相同输入和相同配置可以命中缓存。
9. 在 trace 或独立 LLM call log 中记录 provider、model、耗时、缓存状态、重试次数和错误摘要。
10. 测试默认使用 mock provider，不依赖真实 API key。

## 3. 非目标

v0.2 不做以下内容：

- 不接入 Boss、智联、牛客、小红书等真实网站。
- 不实现多 provider 智能路由。
- 不实现模型自动降级或复杂 fallback 策略。
- 不实现 streaming。
- 不实现 function calling / tool calling。
- 不实现 embedding、RAG、Memory 或向量数据库。
- 不实现长期会话记忆。
- 不把 LangChain Agent 或 OpenAI Agents SDK 作为 Runtime 核心。
- 不在测试中强制调用真实 LLM API。
- 不把 API key 写入代码、trace、缓存或报告。

## 4. 用户输入

CLI 仍接收一个自然语言目标。

示例：

```bash
python apps/cli/main.py run "成都 AI Agent 2027 秋招"
```

v0.2 仍只要求支持单轮命令式运行，不要求支持持续对话。

## 5. 预期输出

每次运行继续生成 v0.1 的输出文件：

```text
data/runs/<run_id>/state.json
data/runs/<run_id>/trace.json
data/reports/<run_id>.md
```

v0.2 新增或扩展输出：

```text
data/cache/llm/
data/runs/<run_id>/llm_calls.json
```

其中：

- `state.json` 必须包含 LLM 解析后的结构化目标。
- `trace.json` 必须能看出 `parse_goal` 是否使用 LLM、是否命中缓存、是否发生重试。
- `llm_calls.json` 记录本次运行的 LLM 调用摘要，不得包含 API key。
- Markdown report 必须在 `Parsed Goal` 或相邻章节展示结构化解析结果。

## 6. 配置要求

v0.2 必须支持本地开发配置，不要求远程配置中心。

`.env` 最低字段：

```text
OPENAI_API_KEY=
OPENAI_BASE_URL=
OPENAI_MODEL=
```

本地 YAML config 可继续沿用：

```yaml
llm:
  provider: openai_compatible
  base_url: ""
  model: ""
```

配置要求：

- API key 只允许从环境变量或未提交的本地配置读取。
- `.env.example` 可以包含空字段，但不得包含真实 key。
- 当 provider 设置为 `mock` 时，不需要 API key。
- 当真实 provider 缺少必要配置时，CLI 必须返回可读错误。
- 测试必须默认使用 mock provider。

## 7. 核心工作流影响

v0.2 不改变 v0.1 的 LangGraph 主流程节点数量，优先只升级 `parse_goal` 节点内部实现。

主流程保持：

```text
START
  -> parse_goal
  -> plan_tasks
  -> run_mock_tool
  -> verify_result
  -> write_report
  -> END
```

### 7.1 `parse_goal`

输入：

- `user_input`
- LLM 配置
- prompt version
- target schema version

输出：

- `parsed_goal`
- LLM 调用摘要
- 追加 `trace`

要求：

- 默认优先使用 LLM provider 解析。
- LLM 输出必须是裸 JSON，不允许 Markdown 代码块包裹。
- 输出必须通过 Pydantic schema 校验。
- 字段缺失时必须显式填入 `"unknown"`、空列表或 `null`，不得省略必填字段。
- 校验失败时最多重试 1 次。
- 重试 prompt 必须包含失败原因摘要。
- 若真实 provider 调用失败，不得静默降级到规则解析，除非配置显式启用 fallback。
- mock provider 必须能稳定返回合法结构，供测试使用。

### 7.2 `plan_tasks`

输入：

- `parsed_goal`

输出：

- `plan`

要求：

- v0.2 仍默认生成一个调用 `mock_job_search` 的任务。
- 计划参数必须兼容 v0.1 `mock_job_search` 输入字段。
- 若 v0.2 schema 新增字段，不得破坏 v0.1 工具调用。

### 7.3 `write_report`

要求：

- 报告中展示结构化解析结果。
- 报告中展示 LLM provider、model、cache hit、retry count 的摘要。
- 报告不得输出 API key 或完整敏感 header。

## 8. Schema 范围

v0.2 可以选择新增 `SearchGoal`，也可以升级 `ParsedGoal`。无论命名如何，必须至少兼容以下字段：

```json
{
  "role_query": "AI Agent",
  "city": "成都",
  "graduation_year": "2027",
  "recruitment_type": "autumn_campus",
  "keywords": ["AI Agent", "LLM", "智能体"],
  "raw_text": "成都 AI Agent 2027 秋招"
}
```

字段要求：

| 字段 | 类型 | 要求 |
| --- | --- | --- |
| `role_query` | string | 必填；无法识别时为 `"unknown"` |
| `city` | string | 必填；无法识别时为 `"unknown"` |
| `graduation_year` | string | 必填；无法识别时为 `"unknown"` |
| `recruitment_type` | string | 必填；建议值为 `autumn_campus`、`spring_campus`、`internship`、`unknown` |
| `keywords` | array[string] | 必填；可为空数组 |
| `raw_text` | string | 必填；保留原始用户输入 |

可选扩展字段：

- `companies`
- `industries`
- `locations`
- `constraints`
- `confidence`
- `warnings`

扩展字段不得影响 v0.1 的 plan 和 mock tool 运行。

## 9. LLM Provider 范围

v0.2 至少包含两个 provider：

```text
mock
openai_compatible
```

### 9.1 `mock` provider

用途：

- 单元测试。
- 集成测试。
- eval 测试。
- 无 API key 的本地开发 smoke test。

要求：

- 固定返回合法 JSON。
- 可配置返回非法 JSON 或 schema 缺字段，用于测试重试和错误处理。
- 不访问网络。

### 9.2 `openai_compatible` provider

用途：

- 对接 OpenAI-compatible Chat Completions API。
- 支持 OpenAI、DeepSeek、Qwen、OpenRouter 等兼容接口，具体 provider 由 base_url 和 model 决定。

要求：

- 支持 `base_url`、`api_key`、`model`、`timeout`。
- 请求必须要求模型返回 JSON。
- 响应必须保留原始文本用于调试和缓存。
- provider 层不得直接依赖 AgentState。
- provider 层不得写报告或调用工具。

## 10. Prompt Contract

v0.2 必须为目标解析建立版本化 prompt。

最低要求：

- prompt 有明确名称和版本，例如 `goal_parser_v1`。
- prompt 明确要求只输出 JSON。
- prompt 包含目标 schema 的字段说明。
- prompt 明确说明未知字段如何处理。
- prompt 不要求模型编造城市、年份、公司或岗位信息。
- prompt 不输出解释性文字。

失败重试 prompt 必须包含：

- 上一次输出的错误类型。
- schema 校验失败摘要。
- 重新输出完整 JSON 的要求。

## 11. 校验与重试要求

v0.2 必须区分三类失败：

| 失败类型 | 示例 | 处理方式 |
| --- | --- | --- |
| `provider_error` | 网络错误、401、超时 | 记录错误；按配置决定是否重试请求 |
| `json_parse_error` | 输出不是合法 JSON | 触发结构化重试 |
| `schema_validation_error` | 缺字段、类型错误 | 触发结构化重试 |

重试规则：

- 默认最多 1 次结构化重试。
- 重试仍失败时，`parse_goal` 标记失败并写入 `errors`。
- 是否允许 fallback 到 v0.1 规则解析必须由配置显式控制。
- fallback 发生时必须写入 trace 和 report 摘要。

## 12. 缓存要求

v0.2 必须实现本地 LLM 调用缓存。

缓存 key 至少包含：

- provider name
- model
- prompt name
- prompt version
- schema version
- normalized input hash

缓存 value 至少包含：

- raw output text
- parsed JSON
- created_at
- provider
- model
- prompt version
- schema version

缓存要求：

- 默认缓存目录为 `data/cache/llm/`。
- 缓存文件必须使用 UTF-8 JSON。
- 相同输入和相同配置再次运行时应命中缓存。
- cache hit 时不得调用真实 provider。
- API key 不得进入 cache key 或 cache value。
- 必须提供禁用缓存的配置开关。

## 13. Trace 与 Observability

v0.2 必须让 LLM 调用可追踪。

单次 LLM 调用摘要至少包含：

```json
{
  "provider": "openai_compatible",
  "model": "example-model",
  "prompt_name": "goal_parser",
  "prompt_version": "v1",
  "schema_version": "v0.2",
  "cache_hit": false,
  "retry_count": 0,
  "duration_ms": 1234,
  "status": "success",
  "error": null
}
```

要求：

- trace 中可以只放摘要，不放完整 prompt 和完整输出。
- `llm_calls.json` 可以保存更详细的非敏感调试信息。
- 不记录 API key、Authorization header 或完整环境变量。
- 如 provider 返回 token usage，应记录在摘要中；没有 usage 时允许为 `null`。

## 14. 错误处理要求

v0.2 必须延续 v0.1 的错误处理原则：

- 节点异常不能被静默吞掉。
- 异常必须进入 `errors`。
- trace 中必须记录失败节点。
- CLI 退出时必须返回可读错误信息。
- 若 `parse_goal` 失败且未启用 fallback，则流程失败。
- 若启用 fallback 且规则解析成功，流程可以继续，但报告必须标记 fallback。

## 15. 文件与目录要求

v0.2 实现建议新增以下文件或目录：

```text
src/campus_job_agent/llm/
src/campus_job_agent/llm/base.py
src/campus_job_agent/llm/mock.py
src/campus_job_agent/llm/openai_compatible.py
src/campus_job_agent/llm/cache.py
src/campus_job_agent/llm/config.py
src/campus_job_agent/prompts/
src/campus_job_agent/prompts/goal_parser.py
src/campus_job_agent/schemas/llm.py
src/campus_job_agent/schemas/goal.py
```

可按 RFC 调整最终文件结构，但必须保持职责边界：

- `llm/` 只负责模型调用、缓存和 provider 抽象。
- `prompts/` 只负责 prompt 构造和版本。
- `schemas/` 负责 Pydantic 校验对象。
- `agent/planner.py` 或等价模块负责把结构化目标接入 workflow。

## 16. 测试要求

### 16.1 单元测试

至少覆盖：

- mock provider 返回合法 JSON。
- mock provider 返回非法 JSON 时触发重试。
- schema validation 成功。
- schema validation 失败。
- cache key 对 provider、model、prompt version、schema version 和输入敏感。
- cache hit 时不调用 provider。
- 配置缺失时返回可读错误。

### 16.2 集成测试

至少覆盖：

- 使用 mock provider 跑通完整 graph。
- 输入 `"成都 AI Agent 2027 秋招"` 后生成结构化目标。
- 输出 `state.json`、`trace.json`、`llm_calls.json`、Markdown report。
- trace 中包含 LLM 调用摘要。
- 第二次相同输入运行可以命中缓存。

### 16.3 Eval 检查

v0.2 eval 可以继续使用规则检查，不要求真实模型参与。

最低 eval：

- `llm_provider_configured = true`
- `structured_goal_valid = true`
- `json_parse_error_captured = true`
- `schema_validation_error_captured = true`
- `retry_attempted = true`
- `cache_hit_visible = true`
- `api_key_not_logged = true`

## 17. 验收标准

v0.2 完成必须满足：

1. 可以通过 CLI 跑通一次完整流程。
2. mock provider 下测试稳定通过。
3. 配置真实 OpenAI-compatible provider 后，可以把用户目标解析为结构化 JSON。
4. LLM 输出不合规时能够被捕获并触发有限重试。
5. 重试失败时错误进入 `errors` 和 trace。
6. 相同输入、相同 provider、相同 model、相同 prompt version 可以命中缓存。
7. `state.json` 包含结构化目标。
8. `trace.json` 或 `llm_calls.json` 包含 provider、model、cache hit、retry count、duration 等摘要。
9. Markdown report 展示结构化解析和 LLM 调用摘要。
10. API key 不出现在 state、trace、cache、report 或测试快照中。
11. 单元测试通过。
12. 集成测试通过。
13. eval 检查通过。

## 18. 后续扩展接口

v0.2 需要为后续版本保留扩展点：

- v0.3：工具输出结构化抽取可以复用 LLM Provider 和缓存。
- v1.0：真实岗位检索后的岗位字段归一化可以复用 JSON-only prompt 和 schema 校验。
- v1.1：经验帖总结可以复用 prompt version、重试和 trace。
- v1.2：岗位要求聚类和最小能力包生成可以复用结构化输出机制。
- v1.3：`schema_valid_rate`、LLM 错误率、缓存命中率进入 eval report。
- v2.0：embedding provider 应独立设计，不与 chat completion provider 强耦合。

## 19. 未决问题

以下问题不阻塞 v0.2 需求确认，但需要在 RFC-0002 中继续明确：

- 是否采用官方 `openai` Python SDK，还是使用 `httpx` 手写 OpenAI-compatible client。
- `SearchGoal` 是否替换 `ParsedGoal`，还是与 `ParsedGoal` 并存。
- LLM cache 使用单文件 JSON、按 hash 分文件，还是 SQLite。
- `llm_calls.json` 是否作为 AgentState 字段同步保存。
- fallback 到 v0.1 规则解析是否默认关闭。
- token usage 的字段格式是否需要跨 provider 统一。
- prompt 文件使用 Python 常量、Markdown 模板，还是 YAML 模板。
