# v0.5 来源与开源项目可行性报告

状态：P0 静态/离线门禁与 P1 浏览器来源 smoke 完成，Ready for Implementation  
日期：2026-07-19  
关联需求：`docs/03_requirements/v0.5-role-profile-graph.md`  
来源清单：`third_party/reference_sources/manifest.yaml`

## 1. 目的

在实现 v0.5 SourceAdapter 前确认：

1. 首版实际覆盖哪些第三方招聘、企业官网和经验来源；
2. 哪些 GitHub 项目可以包装、只能参考或必须拒绝；
3. 企业官网通用采集的确定性能力能覆盖到哪里；
4. 登录态来源可行性与实现后 adapter 验收之间的边界。

本报告不把“仓库能下载/编译”或“浏览器页面可访问”视为“adapter 已实现”。开源项目
只有同时通过许可证、安全边界、结构化输出、raw-before-parse 适配和最小 adapter
live smoke，才可进入 `adopt`；P1 浏览器来源 smoke 只解除代码实现前的来源不确定性。

```text

  → FieldResolution
  → Role Claim / RoleProfile

nowcoder_experience
  → experience raw evidence
  → scoped hiring signals
```

## 3. 准入规则

每个上游项目检查以下字段：

- repository、固定 commit、license；
- 最近维护状态、测试和可运行入口；
- 是否要求登录、如何保存 Cookie/Token；
- 是否默认绕过验证码、robots、限流或风控；
- 是否输出稳定 JSON/错误语义；
- 是否能在解析前保存 raw response；
- 是否可以放在本项目 SourceAdapter 后方；
- offline smoke 和 live smoke 结果。

状态定义：

- `adopt`：允许作为 adapter 依赖候选；
- `blocked_by_auth`：静态/离线检查通过，等待正常登录 live smoke；
- `reference_only`：只学习架构/解析方式，不复制或直接依赖；
- `rejected`：违反许可证、安全、维护或证据边界。

开源许可证只决定代码使用权限，不等于目标网站授权采集。

## 5. 关键发现

### 5.1 BOSS 来源链可用，但候选 CLI 仍需实现 credential 接入

`boss-agent-cli` 的 schema 驱动、JSON envelope、平台注册表、错误恢复和默认只读模式
适合包装为 adapter backend。离线能力已经验证，未登录时也能稳定返回机器可读
`AUTH_REQUIRED`。

2026-07-19 在用户正常登录的 Chrome 中完成了 `AI Agent` 搜索和岗位详情 smoke，
能够读取真实搜索结果、职位状态、薪资、学历、职责、要求、公司、地点和更新时间。
因此 BOSS 作为首版发现来源的浏览器可行性通过。

但该登录态没有导出给 `boss-agent-cli`，也没有读取 Cookie。该候选项目自身仍返回
`AUTH_REQUIRED`，所以 manifest 中保持 `blocked_by_auth`，不能写成 `adopt`。

### 5.2 现有牛客项目不适合直接并入

本轮找到的传统牛客爬虫大多长期未维护。较新的 OfferClaw：

- 没有许可证；
- 通过搜狗发现牛客 URL，本次查询返回空结果；
- 没有 raw-before-parse；
- 页面拿不到点赞数时会生成“质量代理值”，这在统一证据层中属于伪造字段；
- 无法提供 SourceRunReceipt、授权状态和可重放 raw。

因此只能参考 selector/查询词，`nowcoder_experience` 应由本项目按正常登录、
Copy as cURL、raw archive 和 strict JSON contract 自建。

### 5.3 “通用官网爬虫”必须拆成分层能力

离线测试证明：

- Extruct 可以稳定完成标准 JSON-LD 提取；
- Trafilatura 可以抽取正文，但不能保证去除所有导航噪声；
- Ever Jobs 证明 ATS registry 与 JSON-LD-first 具有工程可行性；
- Crawl4AI 能力广，但浏览器、stealth 和依赖范围过重，不适合作为 v0.5 默认基础。

因此 `official_careers` 不采用“一个通用爬虫覆盖所有网站”，而采用：

```text
bounded HTTP fetch + immutable raw archive
  → JSON-LD JobPosting
  → registered ATS/site adapter
  → deterministic DOM/text extraction
  → LLM strict JSON extraction
  → adapter_required
```

最后的 `adapter_required` 是合法结果。LLM 不生成并执行新爬虫代码。

### 5.4 MCP 是可选 transport，不是证据层

上游 CLI、MCP 或 Python 库都只能位于 SourceAdapter 后方。无论底层使用哪种 transport：

- raw bytes 必须先进入本项目 Evidence Store；
- ToolResult 只返回本项目 schema；
- credential 只能以 ref 进入 Graph；
- 上游项目不能直接写画像、checkpoint 或业务数据库。

### 7.1 官网核验案例

BOSS 候选岗位为京东科技 `ai agent开发`，面向 2026 届，BOSS 页面显示“招聘中”，
页面更新时间为 2026-06-19。京东校招官网显示 2027 校招正在进行，但用 `AI Agent`
检索没有对应岗位。

因此：

- 官网能够确认企业招聘站和当前校招批次，但不能确认同一岗位；
- 核验状态为 `official_not_found`，不生成 confirmed identity link；
- BOSS 记录保留，不能仅因官网未找到而判定虚假；
- 画像必须显示来源时间、毕业批次和官网核验状态。

### 7.2 P1 与实现后 live smoke 的边界

P1 已证明三类来源在正常登录和公开官网条件下可访问、可搜索、可读取和可核验。
它没有证明尚未实现的 SourceAdapter 已满足 raw-before-parse。

代码完成后仍需：

1. 三个 adapter 分别执行 opt-in live smoke；
2. 原始响应在解析前进入 Evidence Store；
3. 生成不含凭据的 SourceRunReceipt；
4. 验证 credential ref、auth interrupt/resume、分页、限流、幂等和脱敏；
5. 运行至少一个自动化官网核验案例。

## 8. 当前结论

最新架构已经通过实现前来源门禁：

- 文档架构：通过；
- 开源候选下载与 P0 静态/离线评估：通过；
- 企业官网确定性解析方向：通过，明确接受 `adapter_required`；
- BOSS 浏览器来源 smoke：通过；
- 牛客浏览器来源 smoke：通过；
- 第三方岗位到企业官网的真实二次核验：通过，得到预期的 `official_not_found`。

因此 v0.5 状态为 Ready for Implementation，可以启动 schema、fixture、Graph 和真实
adapter 开发。`boss-agent-cli` 项目本身仍因未接入 credential 而保持
`blocked_by_auth`；代码实现后的 adapter live smoke 仍是版本标记 Implemented 的必要条件。
